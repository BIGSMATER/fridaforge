# M1 回顾：值得复用的模式与避坑清单

> 本文件记录 M1 执行过程中验证有效的模式和暴露的问题，供 M2+ 参考。

## 有效模式

### 1. 即讲即写（每个 Phase 交替教学与编码）

每个 Phase 先讲该阶段涉及的新概念（独立迷你示例），再写代码，补充项目真实代码示例到教学文档。学员困惑在代码层面当场暴露并当场解决。

对比"全部讲完再全部写"：后者学员看完一套语法后没有实际代码对照，理解停留在表面——直到写代码时才暴露盲区。

### 2. 教学文档随代码增长

核心概念（基础语法、错误处理、cobra 命令树等）先写初版。深入内容（逃逸分析、nil 接口陷阱、`var _ =` 编译时检查、context、tabwriter 等）随实现逐个 Phase 补充。

### 3. 项目代码作为教学材料的主体

独立迷你示例（附录 A）用来建立概念框架。但教学正文优先用项目真实代码——学员的目标是看懂 FridaForge 代码，不是看懂脱离上下文的玩具例子。

### 4. 两次 analyze

- 第一次（实现前）：交叉验证 spec.md / plan.md / tasks.md，M1 发现 8 个不一致
- 第二次（实现后）：交叉验证代码 vs 文档，M1 发现 2 个代码问题（Go doc 缺失 + main 包测试缺失）

两次 analyze 各抓不同层次的问题——第一轮抓设计矛盾，第二轮抓实现偏离。

### 5. 接口讲授用"合同隐喻"

M1 讲授接口概念的过程：
- 第一次："鸭子类型" → 学员不感冒
- 第二次："隐式满足" → 学员仍然困惑
- 第三次："合同" + FridaForge M1/M2 替换场景 → 学员懂了

教训：抽象概念需要**两种讲授方式**——先给比喻建立直觉，再用项目代码锚定语境。

### 6. 同文件 Task 合并 Commit

同文件中多个 Task（如 T007+T008 都在 main.go，T014+T015 都在 spec.go）合并为一个 commit。拆开 commit 意味着第一个 commit 是一个不完整的文件，比合并更危险。

## 避坑清单

1. **不要在学员没确认时自顾自推完所有 Task**。M1 犯了——用户说了"开始"后连续推进 26 个 Task。正确做法：每个 Phase 结束等确认。
2. **记得产出教学文档再进入 implement**。M1 第一轮忘了——用户指出后才补。
3. **"计划了但没用"的依赖要在教学文档里解释原因**。viper 在 milestones.md 写了但代码没用——M2+ 要在教学文档里解释为什么。
4. **milestones.md 的目标产出在实现后要更新**。spec.md 可能产生原计划里没有的新 module（如 M1 的 pkg/device/）。
5. **`/speckit.implement` 前的宪法检查要实际到位**。接口命名（DeviceManager → DeviceLister）如果在设计阶段就对照宪法 2.2，analyze 阶段就不会发现 CRITICAL 问题。
